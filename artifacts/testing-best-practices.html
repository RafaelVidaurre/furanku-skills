01 / Evidence model

A test earns trust six ways.

Quality is a balance, not a slogan. More realism can cost determinism; more isolation can cost fidelity. Make the trade-off visible.

01

Relevance

Protect behavior a caller, user, or operator actually relies on.

02

Sensitivity

Fail when the plausible regression it targets is present.

03

Determinism

Give the same result for unchanged code, inputs, and declared environment.

04

Diagnosis

Identify the broken guarantee and provide enough context to begin repair.

05

Resilience

Survive behavior-preserving refactors and harmless orchestration changes.

06

Economy

Return confidence worth its runtime, infrastructure, and maintenance cost.

02 / Operating loop

One repeatable process. No imposed methodology.

6

Each step ends at a checkable boundary. Project conventions decide the framework, sequencing, and style.

01

Frame the evidence

Name the behavior, target failure, observable outcome, and controllable boundary.

Complete when the evidence fits “Given state, when action, then observable outcome.”
02

Choose the smallest credible test

Include only participants needed to expose the risk. State every substitution and fidelity gap.

Complete when included and omitted participants each have a reason.
03

Select cases by risk

Use representative cases, meaningful partitions, boundaries, failures, regressions, and invariants where they change the evidence.

Complete when removing a case would remove a distinct contract or risk.
04

Make the test trustworthy

Keep decisive setup, action, and expectation visible. Control uncertainty and isolate mutable state.

Complete when the test repeats, survives refactors, and explains its failure.
05

Challenge the signal

Run the focused suite and establish sensitivity through an observed failure, safe perturbation, mutation, or causal review.

Complete when intended behavior passes and the target defect credibly fails.
06

Leave the suite healthier

Keep scope, speed, ownership, and maintenance cost proportional to the evidence gained.

Complete when no unexplained flake, hidden shared state, or duplicate evidence remains.

04 / Test craft

Readable at rest. Useful in failure.

A test is executable documentation only when its cause, action, and expected outcome remain visible.

Protect behavior

  • Use the supported interface of the tested unit—not its private assembly.
  • Assert return values, exposed state, domain events, and meaningful side effects.
  • Use interaction checks when the interaction itself is the contract.
  • Keep one coherent behavior per test; use as many assertions as that outcome needs.
  • Derive expected values independently with straight-line, inspectable logic.
  • Name the scenario or action and expected result.

Select cases that discriminate

  • Begin with one representative ordinary case.
  • Add equivalence partitions and boundaries where behavior changes.
  • Cover absence, invalidity, and state transitions named by the contract.
  • Preserve the smallest example of every escaped defect.
  • Use properties for broad laws; inspect generator distributions and replay failures.
  • Parameterize shared behavior, not unrelated stories that merely look alike.

05 / Integration seams

Cross a boundary deliberately.

Name the participants, seam, real and substituted behavior, data lifecycle, and failure that the integration test can expose.

Data store / fidelity

Use the real semantics your claim depends on.

Substitute engines can differ in collation, queries, types, constraints, transactions, locking, and vendor behavior. Include the production engine and relevant version when those semantics carry the risk.

ObserveApplication mapping and results through the supported read/write path.
IsolateFresh schema, unique namespace, disposable instance, or fitting transaction boundary.
CaptureEngine version, migration state, configuration, and decisive records.
CompleteEvery store-specific behavior the change relies on is exercised.
Balance

Use enough production fidelity to expose the risk and enough hermetic control to make the result attributable.

Own

Control dependency versions, readiness, data, identifiers, parallel isolation, and cleanup—including partial failure.

Diagnose

Capture the seam, versions, configuration, correlation IDs, redacted traffic, last state, deadline, and reproduction command.

06 / Fidelity

Every double needs a reason.

Choose the highest-fidelity practical collaborator. State what control it buys and what confidence it leaves behind.

01

Real implementation

Best when fast, deterministic, safe, and easy to construct.

Cost: setup or runtime
02

Maintained fake

Reusable lightweight behavior that preserves the contract and is contract-tested.

Cost: drift and ownership
03

Scenario stub

A precise value or rare error that makes a difficult case reachable.

Cost: duplicated assumptions
04

Interaction check

Useful when the call, count, absence, or ordering is itself observable behavior.

Cost: orchestration coupling

Examples tell stories.
Properties test laws.

Use properties for round trips, idempotence, conservation, ordering, monotonicity, model equivalence, and output validity.

Make the invariant implementation-independent. Inspect generated distributions, preserve seeds, shrink counterexamples, and keep explicit edges that random generation may miss.

07 / Suite health

Trust is a portfolio, not a percentage.

Metrics matter when they change a decision. A single number cannot certify a test suite.

Behavior inventoryWhich important contracts and risks have evidence?
Coverage gapsWhich changed or critical code is never exercised?
MutationWhich plausible faults survive the current assertions?
RuntimeWhich tests control local, presubmit, or release feedback?
FlakinessWhich tests disagree with themselves for unchanged code?
DiagnosisHow quickly can a failure be reproduced and localized?
ChurnWhich tests change during behavior-preserving refactors?
Escaped defectsWhich missing scenario, seam, or level let the bug pass?
01Classify first

Decide whether a failure comes from product behavior, an intentional contract change, the test, the environment, or nondeterminism before editing either side.

02Repair flakes

Capture seed, order, timing, load, versions, and artifacts; remove shared state, guessed sleeps, unowned resources, and work that outlives cleanup.

03Protect feedback

Place evidence in the earliest execution tier where its decision value justifies its cost; track queue time, percentiles, and outliers.

04Prune carefully

Narrow broad regressions when fidelity remains equal, retain distinct seam risks, and remove duplicates or obsolete cases only after their behavior remains protected.
